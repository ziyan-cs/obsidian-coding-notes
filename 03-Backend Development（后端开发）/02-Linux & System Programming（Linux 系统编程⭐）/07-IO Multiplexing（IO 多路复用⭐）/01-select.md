## ⚡ TL;DR（快速决策）

- `select` 的核心是：**让一个线程同时等待多个文件描述符的 I/O 就绪事件**
- 它解决的是“如何不为每个连接单独阻塞等待”的问题
- 若只抓重点，应优先记住：
    - 监听多个 fd
    - 就绪后统一处理
    - 每次调用前要重新设置 fd 集合

## 🧩 Core Idea（核心本质）

- `select` 通过一次系统调用，让进程等待多个 I/O 事件
- 一句话理解：
    - **select 是最经典的 I/O 多路复用接口。**

## 🏗️ 关键特点

1. 使用 `fd_set` 表示监听集合
2. 可监听读、写、异常事件
3. 返回后需要遍历全部 fd 判断谁就绪
4. 存在最大 fd 数量限制等问题

## 🔧 Usage Patterns（可复用理解框架）

1. 用“一个线程盯多个 fd”理解多路复用
2. 用“就绪通知而非完成通知”理解行为
3. 用“每轮扫描所有 fd”理解其开销来源

## ⚠️ Pitfalls（高频错误）

- 忘记每次循环前重置 `fd_set`
- 误以为返回值直接告诉你具体哪些逻辑事件完成
- 忽略 fd 数量上限和线性扫描成本

## 🚀 Performance / Tips（理解深化）

- `select` 适合理解 I/O 多路复用思想，但在大规模连接场景下扩展性有限
- 学 `select` 的重点，是建立“就绪事件驱动”思维模型

## 🧪 Common Scenarios（常见使用场景）

- 小规模网络服务器
- 多输入源等待
- 学习事件驱动 I/O 模型入门
