## ⚡ TL;DR（快速决策）

- 长度固定、连续存储、追求极简访问成本 → 数组
- 需要随机访问 → `a[i]`
- 需要二维表 / 网格 → 二维数组或二维 `vector`
- 大小运行期才确定 → 通常优先 `vector`，不要硬上原生数组
- 题目强调下标、连续区间、原地修改 → 先想到数组模型

## 🧩 Core Idea（核心本质）

- **本质**：数组是一段连续内存上的同类型元素序列
- **关键机制**：元素按下标连续排列，因此可以通过偏移量做 O(1) 随机访问
- **核心优势**：结构简单、访问快、缓存友好，是很多算法题的默认数据载体

## 🔧 Usage Patterns（可复用代码模板）

### 1. 最基础一维数组

```cpp
int a[5] = {1, 2, 3, 4, 5};
```

### 2. 下标访问

```cpp
cout << a[0] << '\\n';
a[2] = 10;
```

### 3. 遍历数组

```cpp
for (int i = 0; i < n; ++i) {
	cout << a[i] << '\\n';
}
```

### 4. 统计和 / 最值

```cpp
int sum = 0;
for (int i = 0; i < n; ++i) {
	sum += a[i];
}
```

### 5. 二维数组

```cpp
int g[3][4] = {0};
g[1][2] = 5;
```

### 6. 函数里处理数组区间

```cpp
int get_sum(int a[], int n) {
	int sum = 0;
	for (int i = 0; i < n; ++i) sum += a[i];
	return sum;
}
```

### 7. 配合 STL 算法

```cpp
sort(a, a + n);
reverse(a, a + n);
```

### 8. 初始化为 0

```cpp
int cnt[100] = {0};
```

## ⚠️ Pitfalls（高频错误）

- 数组下标从 `0` 开始，最容易写成越界
- `a[n]` 是越界，不是最后一个元素，最后一个是 `a[n - 1]`
- 原生数组长度固定，不能像 `vector` 一样动态扩容
- 数组传函数时会退化成指针，函数里拿不到完整长度信息
- 大数组开在局部栈上可能爆栈
- `sizeof(a)` 只有在当前作用域真的是数组时才是整个数组大小
- 多维数组下标顺序写反时，问题很难第一时间看出来
- 原生数组不做边界检查，越界就是未定义行为

## 🚀 Performance / Tips（性能优化）

- 数组非常适合高频随机访问和顺序扫描
- 连续内存通常更缓存友好，遍历效率高
- 静态大小明确时，数组常比更重的抽象更直接
- 需要动态大小时，优先换 `vector`，不要手写不安全逻辑
- 刷题里计数桶、前缀和、差分、双指针，常都直接落在数组上

## 🧪 Common Scenarios（常见使用场景）

- **输入一组数字**：一维数组
- **网格 / 矩阵**：二维数组
- **频率统计**：计数数组
- **前缀和 / 差分**：数组原地或辅助数组
- **双指针 / 滑动窗口**：基于下标处理连续区间
- **排序与二分基础载体**：数组

## 🧾 Minimal Template（最小可运行模板）

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
	ios::sync_with_stdio(false);
	cin.tie(nullptr);

	int a[5] = {3, 1, 4, 1, 5};
	sort(a, a + 5);

	for (int i = 0; i < 5; ++i) {
		cout << a[i] << ' ';
	}
	cout << '\\n';

	return 0;
}
```

## 📌 One-liner Summary（一句话总结）

👉 数组的核心不是“能放很多元素”，而是它提供了 **固定长度 + 连续内存 + O(1) 下标访问** 这组非常适合算法题基础建模的能力。
