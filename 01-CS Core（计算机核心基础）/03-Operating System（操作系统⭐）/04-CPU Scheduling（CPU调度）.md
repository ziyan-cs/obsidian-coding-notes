## ⚡ TL;DR（快速决策）

- CPU 调度的核心是：**决定下一个由谁获得 CPU 执行权**
- 调度目标通常包括：
    - 提高吞吐量
    - 降低等待时间
    - 提高响应速度
    - 保证公平性
- 操作系统不是随机调度，而是依靠调度算法平衡多种目标

## 🧩 Core Idea（核心本质）

- 当多个进程 / 线程都准备运行时，必须由调度器决定执行顺序
- 一句话理解：
    - **CPU 调度就是在有限 CPU 时间上分配执行机会。**
- 调度本质上是资源分配问题

## 🏗️ 常见算法

1. FCFS（先来先服务）

- 简单，但可能造成长任务阻塞短任务

1. SJF / SRTF

- 倾向短作业，平均等待时间较优
- 但实现中需要预测执行时间

1. Round Robin

- 时间片轮转，交互性较好

1. Priority Scheduling

- 按优先级调度
- 可能导致低优先级任务饥饿

## 🔧 Usage Patterns（可复用理解框架）

1. 用“目标不同，算法不同”理解调度

- 交互式系统重响应
- 批处理系统重吞吐

1. 用“时间片”理解多任务体验

- 时间片越短，交互响应更好，但切换成本更高

## ⚠️ Pitfalls（高频错误）

- 只背算法名，不理解它们各自优化什么指标
- 以为存在一个适合所有场景的最优算法
- 忽略时间片、优先级和上下文切换成本的关系

## 🚀 Performance / Tips（理解深化）

- 调度本质是多目标权衡：公平、效率、响应、开销无法同时极致优化
- 理解 CPU 调度后，更容易理解线程池、任务队列、系统负载等实际问题

## 🧪 Common Scenarios（常见使用场景）

- 理解多任务系统响应性
- 分析任务等待和饥饿问题
- 为进程线程管理与并发调优打基础

## 🧾 Minimal Template（最小理解模板）

```
多个可运行任务 -> 调度器选择 -> 获得 CPU -> 时间片结束或阻塞 -> 重新调度
```

## 📌 One-liner Summary（一句话总结）

- **CPU 调度的核心，是在多个可运行任务之间分配有限的处理器时间。**
