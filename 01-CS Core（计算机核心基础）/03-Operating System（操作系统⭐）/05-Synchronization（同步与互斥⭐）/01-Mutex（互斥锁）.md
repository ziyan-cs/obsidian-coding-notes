## ⚡ TL;DR（快速决策）

- 互斥锁的核心是：**保证同一时刻只有一个线程进入临界区**
- 它解决的是“共享资源不能被并发破坏”的问题
- 若只抓重点，应优先记住：
    - lock 获取锁
    - unlock 释放锁
    - 临界区越小越好

## 🧩 Core Idea（核心本质）

- 当多个线程同时修改共享数据时，结果可能不正确
- 互斥锁通过串行化临界区访问来保证一致性
- 一句话理解：
    - **互斥锁就是给共享资源加访问门禁。**

## 🏗️ 关键概念

1. 临界区

- 访问共享资源的那段代码

1. 加锁与解锁

- 进入临界区前加锁
- 离开临界区后解锁

1. 互斥性

- 同一时刻只允许一个执行流持有锁

## 🔧 Usage Patterns（可复用代码模板）

```cpp
mutex m;

m.lock();
// 临界区
m.unlock();
```

更推荐 RAII：

```cpp
lock_guard<mutex> guard(m);
// 临界区
```

## ⚠️ Pitfalls（高频错误）

- 加锁后忘记释放
- 锁粒度过大，导致并发性下降
- 在临界区做耗时操作
- 多把锁顺序不一致，埋下死锁风险

## 🚀 Performance / Tips（理解深化）

- 互斥锁能保证正确性，但可能牺牲并发度
- 正确性通常优先于性能，但锁的范围设计非常重要
- 优化同步时，常需要在“安全”和“并发度”之间权衡

## 🧪 Common Scenarios（常见使用场景）

- 共享计数器
- 共享队列
- 日志写入
- 多线程资源保护

## 🧾 Minimal Template（最小理解模板）

```
共享资源 -> 进入前加锁 -> 独占访问 -> 访问后解锁
```

## 📌 One-liner Summary（一句话总结）

- **互斥锁的核心，是通过独占访问机制保护共享资源不被并发破坏。**
