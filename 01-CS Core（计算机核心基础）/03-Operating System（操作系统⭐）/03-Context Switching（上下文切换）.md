## ⚡ TL;DR（快速决策）

- 上下文切换的核心是：**操作系统暂停一个执行流，保存其现场，再恢复另一个执行流**
- 切换对象可以是线程，也可以是进程
- 上下文切换能实现多任务，但本身会带来开销
- 若只抓重点，应优先记住：
    - 切换要保存状态
    - 切换会耗费时间
    - 切换过多会影响性能

## 🧩 Core Idea（核心本质）

- CPU 某一时刻通常只执行一个线程
- 多任务表象来自操作系统快速轮换执行流
- 一句话理解：
    - **上下文切换就是操作系统在不同执行流之间切换 CPU 使用权。**

## 🏗️ 切换时保存什么

1. 寄存器状态
2. 程序计数器
3. 栈指针
4. 调度信息
5. 进程相关内存映射信息（进程切换时尤其重要）

## 🔧 Usage Patterns（可复用理解框架）

1. 用“保存现场 -> 恢复现场”理解过程
2. 用“切换频繁 ≠ 一定更好”理解代价

- 切换过多时，CPU 时间会消耗在调度而不是有效工作上

1. 用“线程切换 vs 进程切换”理解差异

- 进程切换通常更重
- 线程切换通常更轻，但并非免费

## ⚠️ Pitfalls（高频错误）

- 把并发效果误以为真正同时执行
- 忽略上下文切换成本
- 只会背定义，不理解现场具体包含哪些状态

## 🚀 Performance / Tips（理解深化）

- 线程数并不是越多越好
- 高并发程序若切换过于频繁，性能可能下降
- 上下文切换和缓存失效、调度粒度密切相关

## 🧪 Common Scenarios（常见使用场景）

- 理解多任务系统为何能同时运行多个程序
- 分析线程过多造成的性能问题
- 为 CPU 调度和并发优化打基础

## 🧾 Minimal Template（最小理解模板）

```
暂停当前执行流 -> 保存状态 -> 选择下一个 -> 恢复状态 -> 继续执行
```

## 📌 One-liner Summary（一句话总结）

- **上下文切换的核心，是在保存与恢复执行现场的基础上完成 CPU 对不同任务的切换。**
