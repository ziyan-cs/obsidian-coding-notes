## ⚡ TL;DR（快速决策）

- 死锁的核心是：**多个执行流彼此等待对方持有的资源，导致都无法继续**
- 死锁不是程序“崩了”，而是系统进入互相卡住的等待状态
- 若只抓重点，应优先记住四个必要条件：
    - 互斥
    - 占有且等待
    - 不可抢占
    - 循环等待

## 🧩 Core Idea（核心本质）

- 死锁本质是资源分配关系形成闭环等待
- 一句话理解：
    - **死锁就是等待关系中的环。**
- 只要破坏形成死锁的关键条件之一，就可能避免死锁

## 🏗️ 关键内容

1. 死锁发生条件

- 经典四条件同时满足时，可能发生死锁

1. 处理思路

- 预防：从设计上破坏必要条件
- 避免：动态判断是否进入危险状态
- 检测与恢复：允许发生后再检测处理

1. 常见诱因

- 多把锁嵌套
- 锁顺序不一致
- 长时间占有资源不释放

## 🔧 Usage Patterns（可复用理解框架）

1. 用“资源等待图”理解死锁

- 若等待关系形成环，就存在风险

1. 用“统一加锁顺序”预防死锁

- 所有线程按相同顺序申请锁，可显著降低风险

## ⚠️ Pitfalls（高频错误）

- 只会背四条件，不会分析实际代码场景
- 把普通阻塞和死锁混为一谈
- 忽略多资源竞争中的顺序问题

## 🚀 Performance / Tips（理解深化）

- 死锁问题的重点不是记定义，而是学会从资源依赖关系分析代码
- 工程上常通过约定锁顺序、超时、层级锁等方式降低风险
