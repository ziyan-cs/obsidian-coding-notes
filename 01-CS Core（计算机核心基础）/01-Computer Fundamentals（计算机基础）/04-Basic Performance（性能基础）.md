## ⚡ TL;DR（快速决策）

- 性能基础的本质是：**衡量系统完成任务的速度、效率和资源消耗**
- 讨论性能时，至少要分清：
    - 延迟（Latency）
    - 吞吐（Throughput）
    - 资源利用率
- 性能问题很少来自单点，通常来自系统链路中的瓶颈
- 如果只抓核心，优先记住：
    - 快不快要先定义指标
    - 优化要先找瓶颈
    - 不同场景关注的性能指标不同

## 🧩 Core Idea（核心本质）

- 性能不是单纯“运行时间短”这一个维度
- 一句话理解：
    - **性能分析的核心，是用合适指标刻画系统效率，并找到限制因素。**
- 评价性能时常见问题：
    - 单次任务完成要多久？
    - 单位时间能处理多少任务？
    - 资源是否被有效利用？

## 🏗️ 关键概念

1. 延迟（Latency）

- 一个请求从开始到结束所需的时间
- 更关注单次响应体验

1. 吞吐（Throughput）

- 单位时间能处理多少任务
- 更关注整体处理能力

1. 瓶颈（Bottleneck）

- 系统中限制整体性能的最弱环节
- 优化非瓶颈部分，收益通常有限

1. 局部性（Locality）

- 时间局部性：最近用过的数据，之后还可能再用
- 空间局部性：某个位置附近的数据也更可能被访问
- 局部性直接影响缓存效果

## 🔧 Usage Patterns（可复用理解框架）

1. 先定义性能指标

- 交互式程序更关心延迟
- 批处理系统更关心吞吐

1. 先定位瓶颈再优化

- CPU、内存、磁盘、网络都可能成为瓶颈
- 优化前应先测量，不应只凭直觉判断

1. 用链路思维看性能

- 输入、计算、存储、通信任一环都可能拖慢整体

## ⚠️ Pitfalls（高频错误）

- 不定义指标就讨论“快不快”
- 只优化自己熟悉的部分，而不是实际瓶颈部分
- 忽略 I/O 和内存访问成本，只盯 CPU 计算
- 把个别测试样例当成稳定性能结论

## 🚀 Performance / Tips（理解深化）

- 性能优化的第一原则不是“改代码”，而是“先测量”
- 很多性能问题本质上是数据访问问题，而不是算术运算问题
- 理解性能基础后，缓存、流水线、并行、调度等内容会更自然地串起来

## 🧪 Common Scenarios（常见使用场景）

- 分析程序为什么慢
- 理解响应时间与吞吐量差异
- 理解缓存和局部性价值
- 为体系结构与系统优化打基础

## 🧾 Minimal Template（最小理解模板）

```
定义指标 -> 找瓶颈 -> 测量验证 -> 再优化
```

## 📌 One-liner Summary（一句话总结）

- **性能基础的核心，是用合适指标评估系统效率，并围绕瓶颈进行分析与优化。**
