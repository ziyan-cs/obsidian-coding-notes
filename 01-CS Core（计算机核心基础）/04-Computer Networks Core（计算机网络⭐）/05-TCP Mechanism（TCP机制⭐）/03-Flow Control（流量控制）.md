## ⚡ TL;DR（快速决策）

- 流量控制的核心是：**避免发送方发得过快，超过接收方处理能力**
- 它关心的是接收方是否来得及收，而不是网络是否拥堵
- TCP 主要通过接收窗口机制实现流量控制

## 🧩 Core Idea（核心本质）

- 即使网络通畅，接收方缓冲区也可能来不及处理大量数据
- 一句话理解：
    - **流量控制是发送速率对接收能力的匹配机制。**

## 🏗️ 关键概念

1. 接收窗口（rwnd）

- 接收方通告自己当前还能接收多少数据

1. 发送速率限制

- 发送方根据窗口大小调整未确认数据量

1. 零窗口

- 接收方暂时无可用缓冲区时，可通告 0
- 发送方需等待窗口恢复

## 🔧 Usage Patterns（可复用理解框架）

1. 用“接收方能力限制发送方”理解流量控制
2. 区分流量控制与拥塞控制

- 流量控制：防止压垮接收方
- 拥塞控制：防止压垮网络

## ⚠️ Pitfalls（高频错误）

- 把流量控制和拥塞控制混为一谈
- 只记窗口概念，不理解窗口是在表达接收能力
- 忽略应用处理速度和接收缓冲之间的关系

## 🚀 Performance / Tips（理解深化）

- 流量控制保证的是通信双方速率协调，不是全网性能最优
- 理解接收窗口后，再看 TCP 发送窗口和拥塞窗口会更清晰

## 🧪 Common Scenarios（常见使用场景）

- 理解大文件传输中的发送节奏
- 理解慢接收端为何会限制 TCP 吞吐
- 为 TCP 机制整体学习打基础

## 🧾 Minimal Template（最小理解模板）

```
接收方通告可接收窗口 -> 发送方按窗口发送 -> 避免压垮接收方
```

## 📌 One-liner Summary（一句话总结）

- **流量控制的核心，是根据接收方处理能力限制发送速率，避免接收端缓冲失控。**
