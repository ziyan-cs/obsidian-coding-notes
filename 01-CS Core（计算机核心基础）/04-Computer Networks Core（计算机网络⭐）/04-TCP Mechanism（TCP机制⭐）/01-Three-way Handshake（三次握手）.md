#network #tcp #three-way-handshake #connection-establishment

## ⚡ TL;DR（快速决策）

- 三次握手的核心是：**让通信双方确认彼此的发送与接收能力，并建立初始序列号同步**
- 典型流程：
    - SYN
    - SYN+ACK
    - ACK
- 它不是“形式主义”，而是 TCP 建立可靠连接所必需的状态协商过程

## 🧩 Core Idea（核心本质）

- TCP 是面向连接协议，双方必须先建立共同上下文
- 三次握手解决的是“双方是否都准备好通信”的问题
- 一句话理解：
    - **三次握手是 TCP 连接建立时的双向确认机制。**

## 🏗️ 三步流程

1. 客户端发送 SYN

- 表示希望建立连接，并给出初始序列号

1. 服务端返回 SYN+ACK

- 表示收到请求，并同意建立连接，同时返回自己的初始序列号

1. 客户端返回 ACK

- 表示收到服务端确认，双方进入已建立连接状态

## 🔧 Usage Patterns（可复用理解框架）

1. 用“双向确认”理解三次而不是两次

- 两次只能确认一方收到
- 三次才能确认双方收发链路都基本可用

1. 用“状态同步”理解握手

- 除了问候，更重要的是同步序列号和连接状态

## ⚠️ Pitfalls（高频错误）

- 只背 SYN/ACK 名字，不理解其确认意义
- 误以为两次握手足够
- 把三次握手和四次挥手混淆

## 🚀 Performance / Tips（理解深化）

- 握手带来额外 RTT 成本，但换来可靠连接语义
- 理解三次握手后，更容易读懂连接队列、半连接、SYN flood 等问题

## 🧪 Common Scenarios（常见使用场景）

- 浏览器建立 TCP 连接
- 服务端监听和 accept 流程
- 抓包分析 TCP 建连过程
